# 首启权限引导设计

## 背景

当前应用只在设置窗口中展示权限状态，并在启动时直接触发辅助功能权限请求。对于首次使用者来说，权限需求的原因、完成顺序和完成反馈都不够清晰，也容易出现“权限未完成但应用已经进入正常态”的割裂体验。

参考 Typeless 的权限引导模式，本次设计引入一个独立的首启权限引导窗口，用于在应用首次进入或权限缺失时统一完成系统授权。

## 目标

- 在应用启动时显式引导用户完成 `辅助功能` 与 `输入监听` 授权
- 在两项权限完成前阻止进入正常使用态
- 通过单独窗口提供清晰、可复查的授权状态与操作反馈
- 保持设置窗口继续承担“快捷键与权限状态管理”职责，但不再承担首启门禁职责

## 范围

### 包含
- `辅助功能` 首启强制引导
- `输入监听` 首启强制引导
- 启动阶段门禁与权限完成后的回流逻辑
- 设置窗口中的权限说明与引导入口同步
- 与首启权限引导相关的文档与测试更新

### 不包含
- `提醒事项` 授权纳入首启强制流程
- 多页 onboarding 或账户体系
- 首启完成后的营销页、欢迎页或其它教学流程

## 交互结构

首启权限引导使用独立窗口，采用双栏布局：

- 左侧为任务区
  - 顶部步骤条：`欢迎` → `授权` → `开始使用`
  - 主标题：完成权限授权后即可开始使用
  - 两张权限卡片：
    - `辅助功能`
    - `输入监听`
  - 每张卡片展示：
    - 权限名称
    - 用途说明
    - 当前状态
    - 授权按钮或重新检查按钮
    - 已完成勾选态
  - 底部主按钮：`继续使用 Mac Text Actions`
    - 默认禁用
    - 仅当两项权限均完成时可点

- 右侧为信任说明区
  - 只在用户主动触发 `global shortcut` 后读取 `selected text`
  - 不会在未确认时自动执行 `Replace Selection`
  - 权限缺失时会明确提示，不会静默失败

## 状态机

### 状态来源
- `辅助功能`：通过现有权限状态提供器检查
- `输入监听`：通过现有权限状态提供器检查

### 页面状态
- `待授权`
- `部分完成`
- `全部完成`

### 行为规则
1. 应用启动时先检查两项权限状态
2. 若任一未完成：
   - 显示首启权限引导窗口
   - 不启动 `global shortcut` 监听
   - 不开放工具工作区作为默认入口
3. 用户点击某项 `授权` 后，触发对应系统权限请求
4. 用户返回应用后可点击 `重新检查`
5. 两项完成后解锁底部继续按钮
6. 点击继续按钮时再次校验：
   - 全部完成：关闭引导窗口，进入正常使用态
   - 仍有缺失：停留当前页并提示缺失项

## 启动门禁

门禁收敛在 `AppDelegate`：

- 应用启动时先构建状态栏、主菜单等基础壳层
- 但在权限未完成前，不启动 `KeyboardMonitor`
- 任何试图打开工具窗口或设置窗口的入口，在门禁未解除时都统一拉回首启权限引导窗口
- 权限完成后再启动 `KeyboardMonitor`，恢复正常入口行为

## 设置窗口调整

设置窗口仍保留权限状态展示，但改为服务于“已安装后复查”的场景：

- 显示当前 `辅助功能` / `输入监听` 状态
- 保留“申请授权”“重新检查权限”等入口
- 在文案上补充：首次使用时会先进入权限引导

## 失败反馈原则

- 明确指出缺失的是哪一项权限
- 不把“重新检查”伪装成“继续”
- 不假设用户点击授权后系统一定已经完成设置
- 不允许通过关闭窗口或切换入口绕过首启门禁

## 测试策略

- 为首启权限引导视图模型补充测试：
  - 缺权限时继续按钮不可用
  - 两项都完成时继续按钮可用
  - 点击继续时会进行最终检查
- 为应用启动门禁补充测试：
  - 缺权限时不启动 `KeyboardMonitor`
  - 缺权限时优先进入首启权限引导
  - 权限完成后才启动监听
- 为窗口控制器补充测试，确保独立权限引导窗口可被构建
