# Replace Edit Flow Design

## 背景
当前 `result panel` 中的 `替换` 动作会直接将 `primary result` 写回当前选区，并立即关闭弹框。这种行为无法支持用户在替换前继续编辑，也无法承载“编辑结果后自动重新转换”的交互。

新的目标是将 `替换` 从“立即写回”改为“进入编辑态”，让用户围绕当前转换结果进行二次编辑，并在停止输入后自动重新转换，再由用户显式确认是否写回当前选区。

## 目标
- 点击 `替换` 后，不立即关闭弹框，也不立即写回系统选区
- 弹框进入编辑态，顶部切换为 `当前可编辑值`
- 编辑态默认载入当前 `primary result`
- 原始 `selected text` 保持只读参考
- 用户停止输入约 `300ms` 后自动重新识别和转换
- 时间戳与日期互转场景保留原始精度语义：
  - 原始输入为 `10` 位时间戳，则编辑后的日期再转换为 `10` 位时间戳
  - 原始输入为 `13` 位时间戳，则编辑后的日期再转换为 `13` 位时间戳
- 该交互方式应能被其他支持 `replaceSelection` 的功能复用

## 非目标
- 本轮不把 `result panel` 改造成双编辑器工作台
- 不引入多步骤撤销栈或历史版本浏览
- 不在编辑态中自动写回系统选区

## 交互方案

### 1. 预览态
- 保持当前默认展示
- 展示原始选中文本、自动识别结果和 `secondary action`
- 点击 `替换` 时切换到编辑态

### 2. 编辑态
- 顶部模块标题切换为 `当前可编辑值`
- 编辑器内容初始化为当前 `primary result`
- 下方增加只读的 `原始选中文本` 参考区
- 用户修改编辑器内容后，经过 `300ms` 防抖自动触发重新转换
- 重新转换成功时，结果区展示新的转换结果
- 重新转换失败时，保留编辑内容，并展示错误信息

### 3. 提交动作
- 编辑态中原 `替换` 按钮改为 `应用替换`
- 点击 `应用替换` 时，才将当前编辑值写回系统选区
- 写回成功后关闭弹框
- 写回失败则保留编辑态并提示错误

## 架构设计

### 1. 编辑会话模型
在 app 层引入轻量编辑会话状态，至少包含：
- `mode`: `preview` | `editing`
- `originalSelectedText`
- `editableText`
- `liveResult`
- `transformContext`

### 2. 转换上下文
为支持“自动适配其他功能”，编辑态不要写死时间戳逻辑，而是引入轻量上下文：
- `preserveTimestampPrecision`: `seconds` | `milliseconds` | `none`
- 后续若 JSON、URL 或其他类型需要上下文，也可继续扩展

时间戳上下文的来源：
- 如果初始检测结果是 `timestamp` 且输入长度为 `10`，记录为 `seconds`
- 如果初始检测结果是 `timestamp` 且输入长度为 `13`，记录为 `milliseconds`
- 其他情况为 `none`

### 3. 编辑态转换入口
新增一条面向编辑态的统一转换入口：
- 输入为 `editableText + transformContext`
- 内部仍复用 `ContentDetector + TransformEngine`
- 仅在“日期字符串 -> 时间戳”场景下根据 `transformContext` 决定输出秒级还是毫秒级

## 两种候选方案对比

### 方案 A：在现有 `LiquidGlassPopover` 中加入编辑态
优点：
- 复用现有弹框结构
- 改动集中在 app 层
- 与当前 `result panel` 形态最一致

缺点：
- 需要给弹框补状态管理和防抖逻辑

### 方案 B：点击 `替换` 后跳到独立工具工作区
优点：
- 复杂编辑能力更容易扩展

缺点：
- 打断主流程
- 不符合当前“轻量 result panel”方向

结论：采用方案 A。

## 测试策略
- 为编辑会话状态机补单元测试
- 为时间戳精度继承补核心测试
- 为 `替换` 不再立即关闭弹框补 app 层测试
- 为 `300ms` 防抖后的自动转换补可控时钟或调度测试

## 风险
- `LiquidGlassPopover` 当前是展示型视图，加入编辑态后复杂度会上升
- 时间戳精度继承若写在 UI 层，后续容易漂移，应该尽量收敛到转换辅助层
- 自动转换需要防抖，否则用户输入时会出现频繁闪动
