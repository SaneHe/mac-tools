# Status Menu Mode Switch Design

## 背景
当前 `Mac Text Actions` 的主交互仍以 `global shortcut -> 自动识别 -> result panel` 为核心，状态栏菜单主要承担辅助职责，例如打开设置或退出应用。现阶段虽然已经支持 `selected text` 的自动识别和结果展示，但缺少一个可以让用户显式指定“接下来快捷键应该执行什么功能”的轻量入口。

新的目标是在 macOS 顶部状态栏菜单中增加单层下拉菜单列表，直接展示应用支持的模式与对应快捷键。默认模式为 `自动识别`；当用户改选为某个具体模式后，后续按下 `global shortcut` 时不再执行自动识别，而是直接按该模式处理当前 `selected text`，并弹出对应的 `result panel`。

## 目标
- 在状态栏菜单中新增单层模式列表，不使用分组或二级子菜单
- 菜单至少包含 `自动识别` 与当前 `v1` 已支持的功能模式
- 菜单项名称后直接展示对应快捷键
- 当前默认模式在菜单中以勾选态表示
- 默认模式初始值为 `自动识别`
- 当默认模式为 `自动识别` 时，`global shortcut` 保持现有自动检测流程
- 当默认模式为具体功能时，`global shortcut` 直接按该功能执行，并展示 `result panel`
- 模式切换快捷键仅在状态栏菜单展开时生效，不作为全局快捷键

## 非目标
- 不把状态栏菜单改造成多层功能工作台
- 不新增第二套全局快捷键用于模式切换
- 不在切换模式时直接执行当前功能
- 不把多个功能合并为分组菜单或最近使用区
- 不引入超出 `v1` 范围的新功能模式

## 菜单结构设计

### 1. 菜单形态
状态栏菜单采用单层平铺结构，直接列出所有可选默认模式，不做分组，不引入二级子菜单。用户点击顶部状态栏图标后，应能立即看到当前模式和全部可切换项。

建议菜单顺序如下：
- `自动识别`
- `JSON 格式化`
- `JSON Compress`
- `时间戳转本地时间`
- `日期转时间戳`
- `MD5`
- `创建提醒事项`
- 分隔线
- `设置…`
- `退出`

### 2. 展示规则
- 当前默认模式前显示勾选态
- 菜单项名称后直接显示快捷键
- 快捷键显示应遵循 macOS 菜单的原生表现形式，例如 `⌘1`、`⌘2`
- `设置…` 与 `退出` 保持辅助项身份，不与功能模式混合承担执行逻辑
- 菜单名称与产品术语保持一致，不引入新的别名

## 快捷键模型

### 1. 全局执行快捷键
`global shortcut` 继续作为唯一的全局入口，职责不变，只负责“按当前默认模式执行当前 `selected text`”。它不负责切换模式，也不承担菜单导航职责。

### 2. 菜单内模式切换快捷键
模式切换快捷键仅在状态栏菜单展开时生效，本质上属于菜单命令快捷键，而不是系统级全局快捷键。因此应优先采用 `Command + 数字` 的 macOS 原生样式，而不是 `Control + 数字`。

建议映射如下：
- `⌘1`：`自动识别`
- `⌘2`：`创建提醒事项`
- `⌘3`：`JSON 格式化`
- `⌘4`：`JSON Compress`
- `⌘5`：`时间戳转本地时间`
- `⌘6`：`日期转时间戳`
- `⌘7`：`MD5`

### 3. 切换规则
- 用户点击某个菜单项，或在菜单展开时按下对应 `⌘数字`，即可切换默认模式
- 切换默认模式时，仅更新当前模式状态，不直接执行功能
- 切换后菜单勾选态应立即更新
- 可选地展示轻量提示，例如“当前模式：MD5”，但不弹出 `result panel`

## 执行流程设计

### 1. 自动识别模式
当默认模式为 `自动识别` 时，`global shortcut` 沿用当前主流程：
- 读取当前 `selected text`
- 按既有优先级识别内容类型
- 自动生成 `primary result`
- 展示相应的 `result panel`

识别优先级继续复用现有权威定义：
1. 合法 `JSON`
2. `10` 位或 `13` 位时间戳
3. 可解析日期字符串
4. 普通文本

### 2. 指定模式执行
当默认模式为具体功能时，`global shortcut` 不再参与自动识别分流，而是直接按指定模式执行：
- `JSON 格式化`：直接解析并格式化 `JSON`
- `JSON Compress`：直接解析并压缩 `JSON`
- `时间戳转本地时间`：直接按时间戳规则解析并转换
- `日期转时间戳`：直接按日期字符串规则解析并转换
- `MD5`：直接对原文生成 `MD5`
- `创建提醒事项`：直接进入提醒事项构建流程

指定模式的核心原则是“用户已经明确指定要做什么”，因此应优先执行、优先报错，而不是再次猜测输入类型或退回到自动识别路径。

## `result panel` 行为

### 1. 统一入口
无论来自 `自动识别` 还是具体模式执行，完成处理后都统一进入 `result panel`。本轮不引入静默后台执行，以保留结果可见性、复制结果、替换原文和错误反馈。

### 2. 面板标题语义
为避免混淆，顶部标题应体现“自动识别”与“指定模式”两种来源：
- 自动模式：显示 `自动识别 · JSON`、`自动识别 · 时间戳` 等
- 指定模式：显示 `指定模式 · MD5`、`指定模式 · JSON Compress` 等

这样用户可以明确区分“这是系统判断的结果”还是“这是我主动指定的执行模式”。

### 3. 保留动作
`result panel` 继续保留与结果相关的动作，例如：
- `复制结果`
- `替换原文`
- `关闭`

若结果类型或执行模式允许，也可继续展示特定的相关动作，但不改变“模式由菜单决定、执行由 `global shortcut` 触发”的主模型。

## 提醒事项模式规则
`创建提醒事项` 模式允许做轻量的时间信息带入，但不扩展为自然语言解析器。

建议规则如下：
- 当前 `selected text` 整体优先作为提醒事项标题候选
- 如果文本中能被现有日期能力稳定识别出日期或时间信息，则尝试带入提醒时间
- 如果无法稳定识别时间信息，则仅使用标题，不强行猜测时间
- 不新增“明天下午提醒我”这类自然语言时间解析能力
- 若提醒事项权限缺失，直接在 `result panel` 中显示权限错误与引导

## 错误处理

### 1. 自动识别模式
自动模式继续沿用当前规则，包括：
- 读取失败时的错误提示
- `JSON` 非法时显示 `JSON 校验失败`
- 日期无法解析时按普通文本处理

### 2. 指定模式
指定模式下应采用更明确的失败策略：
- `JSON` 相关模式失败时，不降级为普通文本
- `时间戳转本地时间` 失败时，不退回日期解析
- `日期转时间戳` 失败时，不退回普通文本
- `创建提醒事项` 失败时，直接显示权限或创建失败原因

也就是说：
- `自动识别` 负责“识别并生成默认结果”
- `指定模式` 负责“按用户明确意图执行并反馈结果”

## 方案对比

### 方案 A：单层模式菜单
优点：
- 菜单结构最直接
- 用户点开后立即看到全部可选模式
- 更贴合“状态栏菜单承担轻量控制职责”的方向

缺点：
- 模式数量增长后菜单会拉长
- 自动模式与原 `secondary action` 会并列出现，需要用文案和标题语义区分

### 方案 B：分组模式菜单
优点：
- 语义更整齐
- 更容易扩展

缺点：
- 多一层结构
- 不符合当前希望“不要合并、不做分组”的需求

### 方案 C：最近模式 + 完整菜单
优点：
- 高频切换更快

缺点：
- 复杂度明显提高
- 超出当前 `v1` 轻量菜单目标

结论：采用 `方案 A：单层模式菜单`。

## 需要同步更新的文档
由于当前仓库要求顶部菜单、快捷键职责与权限提示在多个入口文档中保持一致，后续实现时应同步更新至少以下文档：
- `README.md`
- `docs/README.md`
- `docs/design/mac-text-actions-design.md`
- `docs/interaction/mac-text-actions-interaction-flow.md`
- `docs/ui/mac-text-actions-ui.md`

如果实际实现还会改动设置页中的快捷键说明，则还需要同步更新设置相关描述，避免入口文档与实际行为漂移。

## 测试与验证建议
- 验证状态栏菜单能正确显示全部模式与对应快捷键
- 验证当前默认模式勾选态会随菜单选择即时更新
- 验证菜单内 `⌘数字` 仅在菜单展开时有效
- 验证 `global shortcut` 在 `自动识别` 模式下仍走既有自动检测流程
- 验证 `global shortcut` 在指定模式下会直接执行对应功能
- 验证指定模式失败时不会回退到自动识别
- 验证 `创建提醒事项` 模式下的标题带入、时间带入和权限错误展示

## 风险
- 当前文档中“菜单栏仅承担辅助职责”的描述需要调整，否则会与新增模式切换能力冲突
- 原本定义为 `secondary action` 的能力现在可被选为默认模式，需在文档中澄清“默认模式”和“结果内附加动作”是两层概念
- 菜单项数量继续增长时，单层菜单可能需要后续重新评估信息密度
