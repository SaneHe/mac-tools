# 基于提交信息自动打 Tag 并发布 Release 设计

## 背景

当前仓库的 `Release` 工作流只在以下场景触发：

- 推送 `v*` tag
- 手动触发 `workflow_dispatch`

这意味着日常向 `master` 推送提交时，即使提交本身已经明确表达“我要发布一个版本”，仍然需要额外手工创建 tag，发布链路不够顺滑。

## 目标

在保留现有 `v*` tag 推送与手动发布能力的前提下，新增一条自动发布路径：

- 当向 `master` 推送提交时
- 如果最新提交信息包含 `tag:vX.Y.Z` 或 `[tag:vX.Y.Z]`
- 工作流自动在远程创建对应的 Git tag
- 并继续构建产物与发布 `GitHub Release`

## 非目标

- 不支持仅凭单词 `tag` 自动生成随机或时间型版本号
- 不支持在未指定版本号时自动猜测版本
- 不支持从多条提交信息中合并提取多个版本
- 不改变当前未签名产物的打包策略

## 方案对比

### 方案 A：固定格式 `tag:vX.Y.Z`

约定在提交信息中使用 `tag:v0.1.0` 或 `[tag:v0.1.0]`。

优点：

- 规则清晰，误触发概率低
- 易于通过正则稳定解析
- 版本号显式可控，便于追踪

缺点：

- 需要团队遵守固定写法

### 方案 B：只要出现 `tag` 就自动生成版本

优点：

- 使用门槛更低

缺点：

- 版本语义不明确
- 容易误触发
- 与现有 `v*` tag 发布策略不一致

### 方案 C：兼容大量自然语言格式

例如 `release v0.1.0`、`发布 v0.1.0`、`tag v0.1.0`。

优点：

- 书写更自由

缺点：

- 规则复杂
- 维护成本高
- 边界行为更难验证

## 设计结论

采用方案 A。

### 触发规则

`Release` 工作流增加对 `master` 分支 `push` 的监听，但只在以下条件满足时继续发布：

- 事件是 `push` 到 `master`
- 最新提交信息匹配 `tag:vX.Y.Z` 或 `[tag:vX.Y.Z]`

若不匹配，则工作流在早期直接结束，不创建 tag，也不构建产物。

### 版本解析规则

- 优先从最新提交信息中提取第一个匹配项
- 解析后统一得到 `vX.Y.Z` 形式的 tag 名称
- 若提交中没有合法版本号，则跳过发布

### 远程打 Tag 策略

为了避免 “工作流里推送 tag 后，另一个依赖 tag 触发的工作流不一定会被 `GITHUB_TOKEN` 再次触发” 的链路不确定性，本次不依赖二次触发。

改为在同一个 `Release` 工作流中：

1. 解析版本
2. 如有需要，创建并推送远程 tag
3. 使用解析出的 tag 作为 release 元数据继续构建和发布

这样可以：

- 复用当前打包逻辑
- 避免额外新增一个中转工作流
- 保持 tag 推送发布、手动发布、提交信息触发发布三条路径共用同一套构建逻辑

### 幂等性要求

- 若远程 tag 已存在，则不重复创建
- 若 release 已存在，则继续沿用当前逻辑上传或覆盖产物

### 文档同步

需要在开发规范文档中明确新增发布约定：

- `Release` 工作流除了监听 `v*` tag 和手动触发外，也支持 `master` 上的提交信息触发
- 触发格式固定为 `tag:vX.Y.Z` 或 `[tag:vX.Y.Z]`
- 该机制默认只面向 `master` 的推送
