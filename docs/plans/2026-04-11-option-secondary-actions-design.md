# Option 型 Secondary Action 统一交互设计

## 背景

当前仓库存在两套不一致的交互入口：

- 快捷键触发后的 `result panel` 只展示 `复制`、`替换` 等底部动作，没有统一承载带 option 的二次操作。
- 独立工具窗口中，`MD5` 与时间戳相关能力通过输入区 `Toggle` 暴露 option，交互与 `result panel` 分裂。

这会导致相同能力在不同入口中的认知模型不一致，也不利于后续新增需要二次切换的功能。

## 目标

为“带 option 的二次操作”建立统一交互规则，并同时覆盖：

- 快捷键触发后的 `result panel`
- 独立工具窗口

统一规则需要支持后续扩展，让新增带 option 的能力不再重复发明交互。

## 设计结论

### 统一规则

当某个能力存在“对当前结果做可逆切换”的 option 时，底部动作栏遵循以下顺序：

1. `复制`
2. `替换`（仅当当前结果支持替换）
3. `option` 按钮

`option` 按钮始终表示“下一步动作”，而不是当前状态本身。

点击后执行以下行为：

- 使用同一份原始输入重新计算 `primary result`
- 切换内部 option 状态
- 更新结果内容
- 将按钮文案翻转为相反动作

### 当前纳入统一规则的能力

#### MD5

- 默认输出小写
- 底部按钮显示 `转大写`
- 点击后结果切换为大写
- 按钮文案切换为 `转小写`

#### 日期转时间戳

- 默认输出秒级时间戳
- 底部按钮显示 `转毫秒`
- 点击后结果切换为毫秒级时间戳
- 按钮文案切换为 `转秒级`

### 不纳入当前 option 切换的能力

#### 时间戳转本地时间

该能力的输出是本地日期时间文本，不是时间戳数值本身。

因此：

- 仍按输入的 `10` 位或 `13` 位数字自动识别
- 不提供“秒级 / 毫秒级”切换按钮

这样可以避免把“输入位数解释”误设计成“结果 option 切换”，减少歧义。

## 技术设计

### 核心模型

在 `MacTextActionsCore` 中为 `TransformResult` 增加可选的 `option action` 描述，用于表达：

- 当前是否存在 option 型二次操作
- 当前 option 的状态
- 当前应显示的按钮文案

同时在 `TransformContext` 中补充可复用的 option 状态，使同一份转换逻辑可以同时服务：

- 快捷键触发后的显式模式执行
- 独立工具窗口中的手动执行
- 未来新增的 option 型动作

### UI 落地

`result panel` 与独立工具窗口都改为读取同一份 option 元数据：

- 有 option 时在底部追加按钮
- 点击按钮后重新计算并刷新当前结果
- 不再在输入区使用 `Toggle` 暴露这类 option

### 未来扩展约束

后续新增功能如果满足以下条件，应复用本规则：

- 存在基于同一输入的可逆切换
- 切换后仍属于同一主能力的结果表达
- 适合作为当前结果的延伸操作，而不是独立模式

例如未来若增加“Base64 标准 / URL Safe”或“十六进制大写 / 小写”等类似能力，应优先接入同一套 option 模型。
